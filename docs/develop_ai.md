# ETA 변동예측 서비스 개발을 위한 AI 대화내용

## 활용 범위

API 검색, 활용 가능한 기술 장단점 비교, Docker 가상환경 시뮬레이션을 통한 로직 오류 검증, 알고리즘, 화면 디자인, 기능 및 문서 개발

## 사용한 도구

### Claude Code

활용 상세:

* **프로젝트 구조 및 Docker 통합 구성**

  * `cargo-tracker` 루트 기준으로 `fe`, `be`, `vessel_route_analyzer`, MongoDB를 통합 관리하도록 Docker Compose 구조 정리
  * BE FastAPI 서버, FE React 서버, MongoDB 컨테이너를 한 번에 실행할 수 있도록 구성
  * MongoDB healthcheck 이후 BE 서버가 실행되도록 의존성 설정
  * 기존 하위 compose 구성을 루트 compose로 이전하고 포트 충돌 문제 정리
  * `.env`, `.gitignore`를 프로젝트 최상단에서 통합 관리하도록 수정

* **백엔드 FastAPI 서버 구조 생성 및 API 구현**

  * `controller`, `service`, `repository`, `dto`, `entity`, `config` 기반의 기본 MVC 패턴 구조 생성
  * Cargo 등록, 조회, 삭제, 수정 흐름에 필요한 API 구조 구현
  * Cargo 데이터의 PK/ID 처리 방식 점검 및 FE로 ID를 함께 전달하도록 수정
  * 날짜만 받던 Cargo DTO를 시간까지 포함하도록 수정하고, 기존 데이터는 `00:00` 기준으로 보정
  * Swagger 문서 구조와 controller 로직을 분리

* **AI 요약 서버 연동 구조 구현**

  * 사용자가 특정 Cargo ID에 대해 요약을 요청하면 BE가 해당 Cargo 데이터를 조회한 뒤 AI 서버에 전달하는 라우팅 구조 추가
  * AI 서버 요청 DTO와 응답 DTO 구조를 실제 JSON 응답에 맞춰 수정
  * AI 서버 응답을 문자열이 아닌 구조화된 객체로 받도록 DTO 클래스 정리
  * AI 서버에서 받은 요약 결과를 FE로 전달하고, Zustand 전역 상태에 저장해 화면에 렌더링하도록 수정
  * 이미 저장된 AI 요약 정보가 있으면 AI 서버를 다시 호출하지 않고 DB의 기존 요약 데이터를 먼저 조회하도록 캐싱 흐름 구현

* **선박 항로 분석 서버 개선**

  * `vessel_route_analyzer`를 독립 서버에서 루트 프로젝트와 연결되는 AI 분석 서버로 통합
  * 항만 좌표 검색, 기상 정보 검색, 기사 검색이 순차 실행되어 느린 문제를 개선하기 위해 병렬 처리 구조로 수정
  * AI 서버 호출 URI, 요청 JSON, 응답 JSON, DTO 구조를 정리
  * 선박 경로 좌표에 단순 위도·경도·시간뿐 아니라 출발지, 경유지, 목적지, 웨이포인트 구분 메타데이터를 추가하도록 구조 개선
  * 출발지·경유지·목적지 좌표에 별도 핀을 표시할 수 있도록 응답 포맷과 FE 렌더링 구조를 함께 수정

* **프론트엔드 API 클라이언트 및 상태관리 구현**

  * React FE에서 API 요청을 담당하는 `api` 객체 구조 생성
  * Cargo DTO를 프론트엔드에서 사용할 수 있도록 정리
  * Zustand 기반 전역 상태관리 구조를 추가해 Cargo 목록, 선택 Cargo, AI 요약 데이터, 경로 데이터를 관리
  * `.env` 기반으로 API 서버 주소를 관리하도록 FE/BE 환경변수 구조 정리
  * BE에서 받은 Cargo ID, 날짜·시간, AI 요약 결과를 FE 컴포넌트와 연동

* **Cargo 입력 폼 및 메인 패널 UI 개선**

  * Cargo 입력 폼을 팝업 방식이 아니라 헤더 아래에 항상 노출되는 가로형 입력 폼으로 변경
  * 프로젝트명, 경로, 품목, 일정 입력 영역을 행 단위로 재배치
  * 날짜와 시간 입력을 함께 다룰 수 있도록 폼 구조 수정
  * Cargo 저장 완료 후 이전 선택 데이터가 계속 보이는 문제를 수정하고, 새로 생성한 Cargo 정보가 메인 패널에 표시되도록 처리
  * Cargo 리스트 아이템 선택 시 해당 Cargo의 상세 정보와 경로가 화면에 반영되도록 수정

* **지도 렌더링 및 경로 시각화 개선**

  * MapTiler / MapLibre 기반 CargoMap 컴포넌트 구조 수정
  * 지도 초기 줌 레벨을 상수화하고, 향후 서버에서 중심 좌표를 받아 동적으로 적용할 수 있도록 개선
  * AI 서버에서 받은 선박 이동 경로 좌표 리스트를 지도 위에 선으로 연결해 표시
  * Cargo 리스트 아이템마다 다른 경로가 표시되도록 store와 지도 렌더링 로직 수정
  * 현재 시각에 가장 가까운 경로 좌표를 계산해 해당 위치에 배 아이콘을 표시
  * 현재 시간이 출발 전이면 출발 지점, 도착 이후이면 도착 지점, 중간이면 가장 가까운 경로 좌표를 기준으로 선박 위치를 표시

* **태풍 경로 및 위험 표시 기능 추가**

  * 태풍 경로 데이터를 지도에 함께 표시하도록 기능 추가
  * 태풍 경로의 ETA 정보와 선박 경로의 `arrive_at` 시간이 겹칠 경우 경고 표시를 할 수 있는 구조 구현
  * 선박 경로와 태풍 경로를 우선 지도에 렌더링하고, 이후 충돌·위험도 연산을 추가할 수 있도록 기반 구조 마련
  * Cargo 아이템 클릭 시 태풍 API를 호출해도 되는지 호출 빈도와 UX 관점에서 검토

* **프론트엔드 레이아웃 개선**

  * 메인 패널을 좌우 6:4 구조로 분할
  * 왼쪽에는 기존 Cargo 정보, 오른쪽에는 새로운 콘텐츠 영역을 배치할 수 있도록 구조 변경
  * 중앙 구분선을 기준으로 정보가 몰리지 않도록 여백과 flex 레이아웃 수정
  * ATD, ETA, ATA 정보가 화면 확대 시 접히거나 사라지는 문제를 수정
  * 지도 높이가 정적으로 고정된 문제를 개선하고, 푸터 위 영역까지 동적으로 확장되도록 레이아웃 조정

* **삭제·수정 기능 및 사용자 인터랙션 개선**

  * Cargo 리스트 아이템 클릭 시 선택된 Cargo ID를 확보하도록 처리
  * Footer에 삭제 및 수정 버튼을 추가
  * 삭제 버튼 클릭 시 확인 alert를 띄운 뒤 DELETE API 호출
  * 삭제 후 Cargo 리스트와 Zustand store가 갱신되도록 수정
  * 선택된 Cargo의 기존 AI 요약 정보가 있으면 화면에 바로 렌더링되도록 처리

* **오류 분석 및 디버깅 지원**

  * Vite에서 `@maptiler/sdk/style.css` import 실패 문제 분석 및 의존성 반영 문제 수정
  * Docker Compose build 이후에도 패키지 누락이 발생하는 원인 점검
  * AI 서버 연결 실패, ASGI 500 오류, httpx 연결 오류 등 백엔드 런타임 오류 분석
  * MongoDB에 저장된 AI 요약 데이터 구조와 ID 구성 확인
  * Windows 배치 파일 한글 인코딩 깨짐 문제를 확인하고 실행 가능한 스크립트 형태로 수정
  * Node/NPM 버전 의존성 문제 가능성 점검

* **데이터 저장 및 재사용 구조 개선**

  * AI 요약 결과를 Cargo ID 기준으로 MongoDB에 저장하고 재사용하는 구조 검토
  * Cargo별 경로 데이터와 요약 데이터를 분리해 관리
  * FE에서 mock 데이터 대신 실제 BE/AI 서버 응답을 사용하도록 전환
  * 선택 Cargo에 대한 요약 데이터가 store에 없으면 먼저 BE 서버에 기존 요약 존재 여부를 조회하도록 흐름 개선

### ChatGPT

활용 상세:

* **프로젝트 구조 및 개발 환경 구성 지원**

  * `fe`, `be`, `ai` 형태의 프로젝트 루트 구조 검토
  * Docker Compose 기반 React, FastAPI, MongoDB 실행 구조 점검
  * Docker 빌드/재빌드 기준, 컨테이너 내부 파일 변경 시 반영 방식 정리
  * GitHub 저장소명, 루트 폴더명 변경 및 프로젝트 공개 설정 방법 안내

* **지도 및 경로 시각화 기능 검토**

  * MapTiler, MapLibre 기반 지도 표시 방식 검토
  * 지도 초기 포커스, bbox 설정, 전체 지도 로딩 후 특정 영역 중심 이동 방식 정리
  * 선박 출발지·목적지·경유지 기반 예상 항로 시각화 구조 검토
  * 한국 → 케냐, 한국 → 마닐라 등 선박 이동 시 예상 경유 국가 및 항로 흐름 분석

* **기상·해양·태풍 API 조사 및 적용 방향 도출**

  * Open-Meteo Marine API를 활용한 특정 좌표 기준 해양 예보 조회 방식 정리
  * `latitude`, `longitude`, `current`, `hourly`, `forecast_days`, `timezone` 등 주요 쿼리 파라미터 의미 정리
  * 전 세계 태풍 경로 및 현재 위치 표시를 위한 Esri ArcGIS API 후보 비교
  * 태풍 현재 위치 API와 예측 경로 API를 분리해 지도에 표시하는 구조 제안

* **선박·물류 데이터 구조 설계 지원**

  * 선박명, 출발지, 도착지, ETA, ATA, 현재 좌표, 예상 경로 좌표 리스트 등 필요한 데이터 항목 정리
  * Shipment No, B/L No, Invoice No 차이 설명
  * 선박 현재 위치, 항로, 컨테이너 ID 추적 가능 범위 검토
  * 항로 네트워크 데이터를 서버가 보유하고 출발지·목적지 기반으로 예상 항로를 매칭하는 현실적인 구조 제안

* **AI 요약 API 설계 검토**

  * 화물 출발지, 경유지, 목적지, 출발 시각, ETA, ATA를 기반으로 AI 요약 요청 JSON 구조 검토
  * AI 응답으로 경로 좌표, 도착 예정 시각, 요약 문구 등을 반환하는 구조 제안
  * 실제 물류 데이터와 AI 추정 데이터를 구분해야 하는 지점 검토

* **오류 분석 및 로그 기반 성능 파악**

  * FastAPI / Codex CLI / 분석 파이프라인 로그를 기반으로 처리 단계별 흐름 분석
  * 국가 추정, AI 실행자/검수자 파이프라인, 응답 파싱 단계의 처리 시간 파악
  * 로그를 통해 병목 구간과 개선 가능 지점 도출
  * 서비스 응답 시간이 길어지는 원인 후보와 해결 방향 정리

* **프론트엔드 오류 해결 지원**

  * Vite React에서 `@maptiler/sdk/style.css` import 실패 원인 분석
  * 패키지 설치 여부, import 경로, Docker 빌드 캐시, node_modules 반영 문제 점검
  * 로컬 실행과 Docker Compose 실행 간 의존성 반영 차이 설명

* **백엔드·DB 개발 지원**

  * PostgreSQL / MongoDB 컨테이너 접속 방법 정리
  * DB 사용자명, 환경변수, 컨테이너명 기준으로 접속 오류 원인 분석
  * React ↔ FastAPI 간 CORS 설정 문제 해결 방향 제시

* **프로젝트 기획 및 기술 선택 보조**

  * 무료 또는 공개 API 기반으로 구현 가능한 범위와 한계 정리
  * 실제 선박 실시간 위치 API, 항로 추정, 해양 날씨 데이터, 태풍 데이터의 역할 구분
  * 단순 AI 응답이 아니라 외부 API + 자체 항로 추정 + 지도 시각화를 조합하는 구조 제안
  * 사내 AI 경진대회 주제로 카고 프로젝트를 발전시키는 방향성 검토
