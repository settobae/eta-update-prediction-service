# ETA 변동예측 서비스 개발을 위한 AI 대화내용

## 활용 범위

API 검색, 활용 가능한 기술 장단점 비교, Docker 가상환경 시뮬레이션을 통한 로직 오류 검증, 알고리즘, 화면 디자인, 기능 및 문서 개발

## 사용한 도구

### Claude Code

활용 상세:

- **인프라 / Docker Compose 구성**
  - 루트 `.gitignore`가 하위 `be/.gitignore`까지 다 커버하도록 통합 요청
  - `be`만 별도 compose로 두면 서버 인스턴스·MySQL을 한 번에 못 다루는 문제 지적 → 상위(root) 레벨 Docker Compose로 통합 요청
  - FE, vessel_route_analyzer(AI 서버) 등 하위 서비스가 늘어날 걸 대비한 compose 구조(하위 compose 포함 방식 vs 최상단 통합 방식) 논의
  - Spring 서버 Dockerfile 작성 및 compose 등록, MySQL 기동 이후 순서로 뜨도록 `depends_on` 조정
  - `docker ps` 결과에서 컨테이너 이름에 `-1`이 붙는 이유 질문
  - vessel_route_analyzer를 독립 서버에서 루트 compose로 편입, 포트 충돌(8000) 해소 위해 8001로 변경, `.env`/`.gitignore`도 루트에서 통합 관리
  - Windows 배치 파일(`start_service.bat`) 인코딩 깨짐, Codex CLI 자동 설치 실패 등 배포 스크립트 트러블슈팅

- **백엔드 아키텍처 설계**
  - `be` 폴더에 FastAPI 기반 MVC(Controller-Service-Repository) 패턴 스캐폴딩 요청 → 이후 "만들란 것만 만들고 나머진 지워" 라며 스코프 축소 지시
  - AIS 소켓 수신부를 "연결 매니저"와 "실제 호출 서비스"로 계층 분리할지 논의, 커넥터 폴더 위치 결정
  - 선박 구독 관리 API를 4개 기능(특정 MMSI 구독/취소, 범위 구독, 전체 취소)으로 단순화하고 컨트롤러 엔드포인트 매칭
  - Cargo 저장 시 PK 존재 여부, id를 read 응답에 포함시킬 수 있는지 질문
  - DTO 필드에 언더바(`_`)가 붙는 이유(pydantic alias) 질문
  - Mongo vs Postgres 중 어떤 DB로 정적/실시간 선박 정보를 저장할지 비교 요청 → 이후 Mongo 기반 Repository로 전환 결정

- **AIS 데이터 파싱 / 실시간 처리 버그**
  - AIS WebSocket은 연결됐는데 실제 데이터가 안 보이는 문제 분석 (원인: raw 로그 미출력, 알 수 없는 필드로 인한 파싱 실패)
  - "모르는 필드가 있어도 파싱이 되어야 하는 것 아니냐"는 방향으로 역질문하며 무시(ignorable) 필드 처리 요구
  - ETA가 `{Day=0, Hour=0, Minute=0, Month=0}` 형태의 이상한 문자열로 오는 문제 → raw 데이터 직접 찍어서 파싱 실패인지 실제 값인지 확인 요청
  - 모든 타임스탬프가 KST(UTC+9)로 통일되어 있는지 검증 요청
  - POST 방식으로 바꾼 컨트롤러 엔드포인트(`trackArea`)가 제대로 수정됐는지 리뷰 요청
  - MongoDB에 예전 스키마로 저장된 데이터의 직렬화 실패(PathPoint, Summary pydantic 에러) 원인 분석 및 ID 불일치 디버깅

- **프론트엔드 — 레이아웃/UX**
  - CargoMainPanel을 좌(6):우(4) 비율로 분할하고 좌측엔 기존 상세 정보, 우측엔 새 컴포넌트(AI 요약)를 배치하는 구조 제안 및 반복 수정
    - 세로 구분선에 컨텐츠가 붙지 않는 문제
    - 화면이 좁아지면 가로 분할 대신 세로 스택으로 전환
    - ATD/ETA/ATA 표시 영역이 화면 확대 시 사라지는 버그
  - 리스트 아이템을 선택했을 때만 패널이 분할되도록 조건부 렌더링 (화물 추가/랜딩 화면에서는 분할 안 되게 수정)
  - 화물 추가 폼: 전체 폭 대신 절반 폭 고정 + 내부 flex, 입력 필드 크기/간격 조정, 취소/저장 버튼 배치 순서에 대한 실무 관례 질문
  - 헤더에 로고 이미지 삽입, 파비콘이 안 바뀌는 문제
  - 새 프로젝트 등록 후 메인 패널에 이전 데이터가 잔존해서 보이는 stale state 버그

- **지도(CargoMap) 기능**
  - 카고 리스트 아이템별 경로가 지도에 그려지긴 하는데 항상 한 가지 경로만 그려지는 버그 리포트
  - 출발지/도착지 외 경유지에는 핀을 찍지 않도록 요청
  - 현재 시각과 가장 가까운 좌표 인덱스에 배 아이콘을 표시하는 로직(시간 보간) 설계 요청
  - 배 아이콘이 스타일 로드 타이밍 문제로 처음엔 보이다가 다른 아이템 클릭 후 안 보이는 버그 → console.log로 원인 추적
  - 배 아이콘 갱신 주기(1시간)를 위한 인터벌 로직 위치 확인
  - 태풍 표시 기능
    - 태풍 관측/예상경로 ArcGIS API 연동, 같은 태풍의 여러 관측점 중 최신 1건만 표시
    - 지나온 궤적과 예측 경로 구분을 위한 화살표 표시 아이디어
    - 태풍 영향 반경 데이터 유무 확인 및 시각화 개선 요청
  - 태풍 경로 ETA와 선박 ETA/좌표가 겹칠 경우 ⚠️ 경고 이모지 표시 요청
  - 경로 좌표에 "출발/경유/도착/웨이포인트" 유형 메타데이터를 추가해 핀을 구분 표시하도록 FE/BE/AI서버 전체 DTO·엔티티·스토어 수정 요청 (레거시 데이터는 메타데이터 없을 시 스킵 처리)

- **AI 요약(항로 위협 분석) 파이프라인**
  - vessel_route_analyzer(FastAPI AI 서버) API 스펙(요청 경로/JSON, 응답 DTO) 정리 요청
  - 좌표 추론을 Codex CLI로 처리하도록 수정
  - BE ↔ AI 서버 간 통신 라우터 URI 수정, 응답 DTO를 문자열 대신 실제 JSON 구조에 맞는 클래스로 재정의
  - AI 서버 요청→파싱→Codex 전송→응답 수신까지 각 단계 로그 추가 요청 (컨테이너 로그로 모니터링)
  - 항만 좌표 검색 / 기상정보 검색 / 기사(뉴스) 검색이 동기 순차 처리라 느린 문제 → 3개 작업 병렬 처리로 리팩터링 요청
  - 기상정보 조회가 느린 원인 분석 요청
  - 뉴스 검색 조건을 "국가명 + 지연/차질/봉쇄/혼잡 키워드"로 좁히고 CNN 등 신뢰 매체로 제한, 검색 시간 제한 및 검색 건수 로깅 요청
  - 500 에러(`httpx.UnsupportedProtocol`, `ReadTimeout` 등) 트러블슈팅
  - AI 요약 캐싱 전략 설계: 스토어에 없으면 → BE에 기존 요약 존재 여부 조회 → 있으면 렌더링, 없으면 빈 화면 + AI 요청 버튼
    - 다른 카고 클릭 시 이전 카고의 "분석중" 메시지가 잘못 유지되는 버그 수정
    - 캐시 우선순위(스토어 → 서버 DB → 없으면 AI 요청) 재확인
  - 위협 분석 호출 시점이 "아이템 클릭마다"가 아니라 필요한 시점에만 되도록 제어

- **검증 요청** (Claude Code에 브라우저로 직접 확인시킨 케이스)
  - Vite dev 서버 구동 후 상세화면 지도 영역이 footer까지 flex로 확장됐는지 스크린샷으로 확인
  - CargoMainPanel 6:4 분할, 구분선, AI 요약 카드 렌더링 여부를 브라우저에서 직접 확인
  - 부산→오사카 mock 경로가 지도에 정상적으로 그려지는지 확인
  - 배 아이콘(🚢) 마커가 실제로 렌더링되는지, 콘솔 에러가 있는지 확인

- **기타**
  - Claude Code `/insights` 사용량 리포트 실행 결과 확인
  - VS Code Claude Code 세션 대화 내보내기(export) 기능 유무 문의

### ChatGPT

활용 상세:

- **프로젝트 구조 및 개발 환경 구성 지원**
  - `fe`, `be`, `ai` 형태의 프로젝트 루트 구조 검토
  - Docker Compose 기반 React, FastAPI, MongoDB 실행 구조 점검
  - Docker 빌드/재빌드 기준, 컨테이너 내부 파일 변경 시 반영 방식 정리
  - GitHub 저장소명, 루트 폴더명 변경 및 프로젝트 공개 설정 방법 안내

- **지도 및 경로 시각화 기능 검토**
  - MapTiler, MapLibre 기반 지도 표시 방식 검토
  - 지도 초기 포커스, bbox 설정, 전체 지도 로딩 후 특정 영역 중심 이동 방식 정리
  - 선박 출발지·목적지·경유지 기반 예상 항로 시각화 구조 검토
  - 한국 → 케냐, 한국 → 마닐라 등 선박 이동 시 예상 경유 국가 및 항로 흐름 분석

- **기상·해양·태풍 API 조사 및 적용 방향 도출**
  - Open-Meteo Marine API를 활용한 특정 좌표 기준 해양 예보 조회 방식 정리
  - `latitude`, `longitude`, `current`, `hourly`, `forecast_days`, `timezone` 등 주요 쿼리 파라미터 의미 정리
  - 전 세계 태풍 경로 및 현재 위치 표시를 위한 Esri ArcGIS API 후보 비교
  - 태풍 현재 위치 API와 예측 경로 API를 분리해 지도에 표시하는 구조 제안

- **선박·물류 데이터 구조 설계 지원**
  - 선박명, 출발지, 도착지, ETA, ATA, 현재 좌표, 예상 경로 좌표 리스트 등 필요한 데이터 항목 정리
  - Shipment No, B/L No, Invoice No 차이 설명
  - 선박 현재 위치, 항로, 컨테이너 ID 추적 가능 범위 검토
  - 항로 네트워크 데이터를 서버가 보유하고 출발지·목적지 기반으로 예상 항로를 매칭하는 현실적인 구조 제안

- **AI 요약 API 설계 검토**
  - 화물 출발지, 경유지, 목적지, 출발 시각, ETA, ATA를 기반으로 AI 요약 요청 JSON 구조 검토
  - AI 응답으로 경로 좌표, 도착 예정 시각, 요약 문구 등을 반환하는 구조 제안
  - 실제 물류 데이터와 AI 추정 데이터를 구분해야 하는 지점 검토

- **오류 분석 및 로그 기반 성능 파악**
  - FastAPI / Codex CLI / 분석 파이프라인 로그를 기반으로 처리 단계별 흐름 분석
  - 국가 추정, AI 실행자/검수자 파이프라인, 응답 파싱 단계의 처리 시간 파악
  - 로그를 통해 병목 구간과 개선 가능 지점 도출
  - 서비스 응답 시간이 길어지는 원인 후보와 해결 방향 정리

- **프론트엔드 오류 해결 지원**
  - Vite React에서 `@maptiler/sdk/style.css` import 실패 원인 분석
  - 패키지 설치 여부, import 경로, Docker 빌드 캐시, node_modules 반영 문제 점검
  - 로컬 실행과 Docker Compose 실행 간 의존성 반영 차이 설명

- **백엔드·DB 개발 지원**
  - PostgreSQL(현재 미사용) / MongoDB 컨테이너 접속 방법 정리
  - DB 사용자명, 환경변수, 컨테이너명 기준으로 접속 오류 원인 분석
  - React ↔ FastAPI 간 CORS 설정 문제 해결 방향 제시

- **프로젝트 기획 및 기술 선택 보조**
  - 무료 또는 공개 API 기반으로 구현 가능한 범위와 한계 정리
  - 실제 선박 실시간 위치 API, 항로 추정, 해양 날씨 데이터, 태풍 데이터의 역할 구분
  - 단순 AI 응답이 아니라 외부 API + 자체 항로 추정 + 지도 시각화를 조합하는 구조 제안
  - 사내 AI 경진대회 주제로 카고 프로젝트를 발전시키는 방향성 검토
